# 请假与考勤 OA（plate-exam）

基于若依 RuoYi-Vue 低代码平台改造的办公自动化场景，覆盖"请假管理 + 考勤管理"，通过工作流引擎串联多角色协作，形成"发起→处理→反馈→归档"闭环。本项目用于《平台软件开发技术》课程考核。

## Language

### 角色

**普通员工**：
系统的申请发起方。可发起请假申请、查询本人申请进度、查看本人考勤记录。
_Avoid_: 用户、申请人、职员

**部门负责人**：
本部门请假申请的审批方。可审批（通过/驳回）本部门请假、查看本部门考勤。通过部门归属识别，而非独立的人员类型。
_Avoid_: 主管、经理、审批人

**系统管理员**：
平台的数据与规则管理方。负责备案归档、管理全量数据、配置规则（请假类型、考勤规则等）。复用若依内置 admin。
_Avoid_: 超级管理员、超管、运维

### 业务

**请假申请**：
由普通员工发起、需经审批流转的一次请假请求，是工作流的核心流转单据。
_Avoid_: 请假单、假条、leave

**考勤记录**：
记录某员工在某一天出勤状态的数据，按"员工×日期"唯一。状态取值为 正常/请假/缺勤/迟到（字典可配）。来源有二：请假归档时系统自动写入"请假"，以及管理员手动维护。本项目不做真实打卡/定位/补卡/排班。
_Avoid_: 打卡记录、出勤单

**请假联动**：
请假申请进入"已归档"时，系统自动把请假日期区间内的考勤记录写为"请假"状态的动作。是请假与考勤两个模块产生数据流的关键环节。
_Avoid_: 同步、回写

### 流程状态

流程状态由字典 `oa_leave_status` 维护（pending/approved/rejected/archived），前端统一用 `dict-tag` 渲染。

**待审批 / pending**：
请假申请已提交、等待部门负责人处理的状态。申请的初始状态（提交即进入，无草稿态）。
_Avoid_: 待处理、未审批

**已通过 / approved**：
部门负责人审批通过、等待管理员备案的状态。
_Avoid_: 已批准、通过

**已驳回 / rejected**：
部门负责人审批不通过的终态。流程就此终止，员工如需请假须新建申请，不在原单据上复活。
_Avoid_: 拒绝、退回、打回

**已归档 / archived**：
管理员完成备案、流程闭环结束的终态。
_Avoid_: 已完成、结束、备案
